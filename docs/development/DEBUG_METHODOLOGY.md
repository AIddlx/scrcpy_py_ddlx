# 调试方法论

本文档总结项目开发中积累的调试经验和方法论，帮助更高效地解决问题。

---

## 核心原则

### 1. 在信息源头处理

**错误做法**：在数据流的末端尝试修复问题
**正确做法**：找到信息已经正确的地方，从那里传递信息

**案例：横屏触摸失效**
- ❌ 在 MCP 服务器的帧发送循环中检测旋转（间接，不可靠）
- ✅ 预览进程已经正确检测旋转，直接发送消息通知 MCP 服务器

```
# 错误思路：依赖复杂的回调链
解码器 → DelayBuffer → 帧发送循环 → 检测旋转 → 更新状态

# 正确思路：在检测点直接同步
预览进程检测到旋转 → 立即发送消息 → MCP 服务器更新状态
```

### 2. 对比工作与不工作的场景

当某个场景不工作，但类似场景工作时，**对比差异是最高效的调试方法**。

**步骤**：
1. 找到一个可以工作的参照物（如 `test_network_direct.py`）
2. 画出两个场景的数据流图
3. 标出差异点
4. 针对差异点分析根因

**案例**：
```
test_network_direct.py (工作):
  video_window._device_size → input_handler._device_size → 触摸事件

MCP 服务器 (不工作):
  preview_process._device_size → (IPC 断裂) → client.state.device_size → 触摸事件
```

差异点：两个 `_device_size` 不同步！

### 3. 使用多 Agent 并行分析

对于复杂问题，组建团队并行分析比单线程调试更高效。

**分工建议**：
- **日志分析 Agent**：搜索日志中的关键信息
- **代码分析 Agent**：追踪代码执行路径
- **流程对比 Agent**：对比工作/不工作场景的差异

**启动方式**：
```python
# 并行启动多个专业 Agent
Task(description="分析日志", subagent_type="bug-analyzer")
Task(description="比较流程差异", subagent_type="Explore")
```

## 调试流程

### 第一步：确认问题范围

1. **问题是否可复现？**
   - 可复现 → 继续分析
   - 不可复现 → 增加日志，等待下次触发

2. **问题发生的条件是什么？**
   - 横屏时？竖屏时？两者都有？
   - 特定操作后？随机发生？

3. **有没有工作的参照物？**
   - 有 → 对比差异
   - 没有 → 从头分析

### 第二步：追踪数据流

1. **画出数据流图**
   ```
   数据源 → 处理A → 处理B → 处理C → 结果
   ```

2. **在每个节点添加日志**
   - 输入是什么？
   - 输出是什么？
   - 状态是否正确？

3. **找出断裂点**
   - 哪个节点的输出不正确？
   - 信息在哪里丢失或变形？

### 第三步：定位根因

1. **检查假设**
   - "这个变量应该被更新" → 真的被更新了吗？
   - "这个回调应该被调用" → 真的被调用了吗？

2. **检查边界条件**
   - 初始状态是否正确？
   - 状态变化时是否正确处理？
   - 并发/竞态条件？

3. **检查信息传递**
   - 跨进程/跨线程通信是否正确？
   - 回调链是否完整？
   - 数据格式是否匹配？

### 第四步：设计修复方案

1. **评估修复位置**
   - 在信息源头修复（推荐）
   - 在中间节点修复
   - 在末端修复（临时方案）

2. **评估修复复杂度**
   - 简单直接 > 复杂间接
   - 主动通知 > 被动检测
   - 单点修改 > 多点修改

3. **考虑副作用**
   - 修复会影响其他功能吗？
   - 需要同步修改其他地方吗？

## 常见陷阱

### 1. 只看症状，不看根因

**错误**：看到坐标不对，就去改坐标转换代码
**正确**：分析为什么坐标不对？是转换逻辑错？还是输入数据错？

### 2. 过度依赖假设

**错误**："这个变量应该被更新了"
**正确**：添加日志验证变量是否真的被更新

### 3. 复杂化简单问题

**错误**：添加复杂的抽象层、回调链
**正确**：最简单的方案往往是最好的

### 4. 忽略跨进程/跨线程问题

**错误**：假设状态自动同步
**正确**：显式同步状态，使用 IPC 消息或信号量

## 实战案例

### 案例 1：横屏触摸失效

**现象**：MCP 预览窗口横屏时触摸不生效，竖屏正常

**分析过程**：
1. 对比 `test_network_direct.py`（工作）和 MCP 服务器（不工作）
2. 发现 `test_network_direct` 的 `input_handler._device_size` 通过回调更新
3. MCP 服务器的 `client.state.device_size` 没有被更新
4. 预览进程已经正确检测旋转，但没有通知 MCP 服务器

**修复方案**：
- 预览进程检测到旋转 → 发送 `device_size_changed` 消息
- MCP 服务器收到消息 → 更新 `client.state.device_size`

**关键教训**：在信息源头直接同步，而不是依赖复杂的回调链

### 案例 2：音频录制音量跳变

**现象**：Opus 格式录音音量随时间增大

**分析过程**：
1. WAV 格式正常 → 问题在 Opus 编码
2. 检查 PyAV Opus 编码器配置
3. 发现 `channels` 属性是只读的，设置失败导致编码器配置不完整

**修复方案**：
- 不手动设置 `channels`，让 PyAV 自动配置
- 使用 `format='flt'` 匹配输入格式

**关键教训**：验证假设（"设置 channels 成功了吗？"）

## 调试工具箱

### 日志搜索命令

```bash
# 搜索关键事件
grep -n "Device rotation\|device_size\|INJECT_TOUCH" log.txt

# 搜索错误和警告
grep -n "ERROR\|WARNING\|Exception" log.txt

# 按时间范围搜索
grep "2026-02-25 00:23:" log.txt | head -100
```

### Python 快速分析

```python
# 分析日志
with open('log.txt', 'r', encoding='utf-8', errors='replace') as f:
    for i, line in enumerate(f):
        if '关键词' in line:
            print(f'{i+1}: {line.strip()}')
```

### 结构化调试

1. **添加调试日志**（不是 `print`）
   ```python
   logger.info(f"[DEBUG] 变量值: x={x}, y={y}")
   ```

2. **标记调试代码**
   ```python
   # DEBUG: 临时添加，问题解决后删除
   ```

3. **保留有用的日志**
   - 关键状态变化
   - 错误恢复
   - 性能指标

## 总结

高效调试的核心：

1. **对比差异** - 找到工作的参照物
2. **追踪数据流** - 找出信息断裂点
3. **在源头处理** - 不要在末端打补丁
4. **简单直接** - 避免过度复杂化
5. **验证假设** - 不要想当然
