# 开发流程规范

本文档定义了 scrcpy-py-ddlx 项目的开发流程，确保修改不会引入意外问题。

---

## 核心原则

1. **文档先行** - 修改代码前先更新设计文档
2. **全局视角** - 考虑修改对所有组件的影响
3. **场景覆盖** - 测试所有可能的使用场景
4. **可追溯性** - 每个修改都要有记录和原因

---

## 修改流程

### 阶段 1: 修改前分析

在修改任何代码之前，必须完成以下检查：

#### 1.1 影响范围分析

```markdown

- 功能：
- 原因：

### 受影响的组件
- [ ] 服务端 (Java)

  - [ ] 控制通道
  - [ ] UDP 发送
- [ ] 客户端 (Python)
  - [ ] 解码器
  - [ ] 解复用器
  - [ ] 渲染器
  - [ ] 控制通道
- [ ] 协议
  - [ ] 消息格式
  - [ ] 包头结构

### 相关文件
列出所有可能需要同步修改的文件
```


### 阶段 2: 设计文档更新

**在修改代码之前**，必须更新相关设计文档：

#### 2.1 协议修改

如果修改涉及通信协议：

```
必须更新：docs/PROTOCOL_SPEC.md
必须检查：docs/development/PROTOCOL_CHANGE_CHECKLIST.md
```

#### 2.2 架构修改

如果修改涉及系统架构：

```
必须更新相关文档：
- docs/development/NETWORK_PIPELINE.md
- docs/development/VIDEO_AUDIO_PIPELINE.md
```

#### 2.3 创建修复文档

对于 bug 修复，创建专门的修复文档：

```
位置：docs/development/known_issues/
命名：ISSUE_NAME_FIX.md
内容：问题现象、原因分析、解决方案、测试结果
```

---

### 阶段 3: 代码修改

#### 3.1 修改原则

1. **最小化修改范围**
   - 只修改必要的代码
   - 避免重构不相关的代码

2. **保持向后兼容**
   - 新功能不应破坏现有功能
   - 添加新参数时提供默认值

3. **添加必要日志**
   - 关键路径添加日志
   - 异常情况添加警告

4. **更新注释**
   - 修改代码时更新相关注释
   - 复杂逻辑添加解释

#### 3.2 服务端修改检查清单

```java
// 修改 MediaCodec 相关代码时
- [ ] 是否考虑了屏幕旋转？
- [ ] 是否考虑了 EOS (End of Stream)？
- [ ] 是否有超时机制？
- [ ] 资源是否正确释放？

// 修改网络相关代码时
- [ ] 是否考虑了网络断开？
- [ ] 是否有重试机制？
- [ ] 是否有超时处理？
```

#### 3.3 客户端修改检查清单

```python
# 修改解码器相关代码时
- [ ] 是否考虑了 CONFIG 包变化？
- [ ] 是否正确处理解码错误？
- [ ] 是否需要重新初始化解码器？

# 修改解复用器相关代码时
- [ ] 是否考虑了包丢失？
- [ ] 是否考虑了乱序？
- [ ] 是否考虑了屏幕旋转后的重新同步？
```

## 常见问题场景参考

### 屏幕旋转

```markdown
## 屏幕旋转影响分析

### 服务端
1. MediaCodec 需要 EOS 并重新配置
2. 新的 CONFIG 包需要发送
3. 虚拟显示需要重建

### 客户端
1. 解码器需要重新初始化
2. CONFIG 数据需要与关键帧合并
3. 渲染器尺寸需要调整

### 检查点
- [ ] CONFIG 包是否正确发送和接收
- [ ] 关键帧是否包含正确的 SPS/PPS
- [ ] 解码器是否正确重新初始化
```

### 网络断开

```markdown
## 网络断开影响分析

### 服务端
1. TCP 控制通道断开检测
2. UDP 数据包发送失败处理
3. 资源清理

### 客户端
1. UDP 接收超时检测
2. TCP 心跳超时检测
3. 自动重连机制

### 检查点
- [ ] 是否有超时机制
- [ ] 资源是否正确释放
- [ ] 重连是否正常
```

## 附录：历史教训

### skipFrames 优化问题

**问题**: 添加 skipFrames 优化时没有考虑屏幕旋转场景，导致 EOS 被跳过，编码器卡住。

**教训**:
1. 优化代码要考虑所有场景
2. 添加新功能时要进行全面的场景测试
3. 复杂优化如果问题多，不如保持简单实现

### CONFIG 合并问题

**问题**: CONFIG 数据与非关键帧合并，导致关键帧缺少 SPS/PPS，视频损坏。

**教训**:
1. 理解 H.264 编码规范（关键帧需要 SPS/PPS）
2. 测试时不仅要看"能播放"，还要测试边界情况
3. 协议相关的修改要特别谨慎
