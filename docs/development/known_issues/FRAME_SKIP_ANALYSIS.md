# Frame Skip 现象分析

## 现象描述

在客户端日志中出现 `[FRAME_SKIP] NV12 frame #xxx skipped (total skips=51)` 警告。

---

## 全局性问题分析

### 核心问题：生产-消费架构设计

```
┌─────────────────────────────────────────────────────────────────┐
│                        数据流架构                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [服务端] ──UDP──→ [Demuxer] ──→ [解码器线程] ──→ [DelayBuffer] │
│    30fps                              30fps        单帧缓冲      │
│                                                     ↓           │
│  [OpenGL Widget] ←──consume()─────────────────────┘            │
│    paintGL: 60fps                                               │
│    实际消费: 30fps                                               │
│    空转: 30fps (50%)                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 问题 #1：paintGL 空转 50%

```
paintGL #61, frames_consumed=29   → 60次调用 / 29次消费 = 50% 空转
paintGL #121, frames_consumed=57  → 60次调用 / 28次消费 = 53% 空转
```

**问题**：
- paintGL 按显示器刷新率（60fps）调用
- 但实际只有 30fps 的帧可消费
- **50% 的 paintGL 调用是无效的空转**

**影响**：
- CPU 资源浪费
- 不必要的函数调用开销

### 问题 #2：单帧缓冲无流量控制

```python
class DelayBuffer:
    def __init__(self):
        self._pending_frame = None  # 只能存一帧！
        self._consumed = True

    def push(self, frame):
        # 无论消费端状态如何，都会覆盖
        previous_skipped = not self._consumed
        self._pending_frame = frame  # 直接覆盖
```

**问题**：
- 缓冲深度 = 1 帧
- 生产端不感知消费端状态
- 任何消费延迟都会导致帧丢失

### 问题 #3：线程间缺乏协调

```
解码线程                          渲染线程（GUI）
   │                                  │
   │ push(frame_1)                    │
   │ ├─→ _pending_frame = frame_1     │
   │                                  │ paintGL()
   │ push(frame_2)                    │ ├─→ consume() → frame_1
   │ ├─→ previous_skipped = True!     │
   │ ├─→ _pending_frame = frame_2     │
   │                                  │ paintGL()
   │ push(frame_3)                    │ ├─→ consume() → None (已消费)
   │ ├─→ previous_skipped = True!     │
   │                                  │
```

**问题**：
- 解码器不知道渲染端的消费速度
- 没有背压（backpressure）机制
- 生产速度 > 消费速度时，帧被丢弃

### 问题 #4：启动阶段的不匹配

| 时间 | 帧号 | 累计skip | 原因 |
|------|------|----------|------|
| 14:28:00.053 | #6 | 1 | **初始化积压**（非拖动） |
| 14:28:00.112 | #19 | 11 | **初始化积压**（非拖动） |

**问题**：
- 系统启动时，解码器先于渲染端就绪
- 11 帧在启动阶段就被 skip
- 这与窗口拖动无关

---

## 窗口拖动只是暴露问题

### 局部 vs 全局

| 视角 | 分析 | 结论 |
|------|------|------|
| **局部** | 窗口拖动导致 skip | "拖动是原因" |
| **全局** | 架构设计导致 skip | "拖动只是触发条件" |

### 全局视角的时间线

```
┌─ 正常运行期 ─────────────────────────────────────────────────┐
│  生产: 30fps  消费: 30fps  状态: 平衡                        │
│  paintGL空转: 30fps (50%)  资源: 浪费                        │
└──────────────────────────────────────────────────────────────┘
                              ↓
┌─ 窗口拖动期 ─────────────────────────────────────────────────┐
│  生产: 30fps  消费: 0fps   状态: 失衡                        │
│  skip: 是      原因: 单帧缓冲无法暂存                        │
└──────────────────────────────────────────────────────────────┘
                              ↓
┌─ 恢复期 ─────────────────────────────────────────────────────┐
│  生产: 30fps  消费: 30fps  状态: 恢复平衡                    │
└──────────────────────────────────────────────────────────────┘
```

**结论**：窗口拖动只是让隐藏的架构问题暴露出来。

---

## 根本原因总结

| 层面 | 问题 | 表现 |
|------|------|------|
| **架构** | 单帧缓冲 | 任何波动都导致 skip |
| **同步** | 无背压机制 | 生产不感知消费状态 |
| **效率** | paintGL 50% 空转 | CPU 资源浪费 |
| **启动** | 初始化顺序不当 | 启动阶段就有 skip |

---

## 改进方向

### 方向 1：事件驱动替代轮询

**当前**：paintGL 轮询 consume()
**改进**：帧到达时触发渲染

```python
# 解码完成后通知渲染
def on_frame_ready(self, frame):
    self._opengl_widget.update()  # 触发 paintGL
```

**效果**：消除 50% 空转

### 方向 2：增加缓冲深度 + 背压

**当前**：单帧缓冲
**改进**：多帧缓冲 + 流量控制

```python
class FrameBuffer:
    def __init__(self, max_size=3):
        self._queue = deque(maxlen=max_size)

    def push(self, frame):
        if len(self._queue) >= self.max_size:
            # 缓冲满，丢弃最旧帧或阻塞生产
            self._queue.popleft()
        self._queue.append(frame)
```

**效果**：吸收短期波动

### 方向 3：生产-消费协调

**当前**：独立运行
**改进**：消费端反馈

```python
# 消费端通知生产端
def consume(self):
    if self._queue.qsize() > threshold:
        self._notify_producer_slow_down()
```

## 数据支撑

### 65 秒内统计

| 指标 | 数值 | 占比 |
|------|------|------|
| 总 paintGL 调用 | ~3900 次 | 100% |
| 实际帧消费 | ~1950 次 | 50% |
| 空转调用 | ~1950 次 | 50% |
| Frame Skip | 51 次 | 2.6% |

### 启动阶段分析

```
14:28:00.053 - frame #6 skipped (初始化)
14:28:00.112 - frame #19 skipped (10帧在59ms内丢失)
```

启动 59ms 内丢失 10 帧，说明生产远快于消费。

---

## 结论

1. **Frame Skip 是架构问题的表现**，不是孤立的 bug
2. **根本原因**：
   - 单帧缓冲无法吸收波动
   - 生产消费缺乏协调
   - 50% paintGL 空转
3. **窗口拖动只是触发条件**，不是根本原因
4. **改进需要架构层面考虑**，而非局部修补
